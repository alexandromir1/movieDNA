# 06. Roadmap

## Принцип

Каждый следующий PR отвечает только на вопрос:

> Поможет ли это игроку захотеть вернуться завтра?

Если нет — задачу откладываем.

Engine, Content Studio и текущая архитектура достаточны для первого публичного релиза.
Приоритет — пользовательский опыт и обратная связь от игроков.

Контент (14–21 Challenge вперёд) — постоянный процесс, не пункт roadmap.

См. также: [PLAYER_JOURNEY.md](./PLAYER_JOURNEY.md).

## Заморозка функциональности (после PR №5)

Не добавляем новые игровые механики, UI-улучшения и рефакторинг «на всякий случай».

**Engine** — стабилен.  
**Content Studio** — MVP.  
**Player Journey** — первая релизная версия.

### Текущий приоритет (soft launch)

1. Только критические баги
2. Поддержка Dev Studio
3. Производство контента (новые Challenge)
4. Расписание минимум на **14–21 день** вперёд
5. GA4 — когда будет Measurement ID
6. Новые идеи — сначала обсуждение, не код

Код — только маленькими изолированными PR, если без него нельзя.

Цель этапа: **первые игроки и данные о поведении**, не рост числа функций.

Фильтр новой идеи: **поможет ли это получить первых игроков быстрее?**  
Если нет — backlog.

## Отложено (после soft launch / 100–500 игроков)

- Percentile / сравнение с игроками
- Auth и sync между устройствами
- Leaderboard
- Push / PWA
- Duplicate Level UI (инструмент Studio; дубликаты **поиска** — см. SL-2)
- Фильтры в Архиве
- Локальная дата вместо UTC (если метрики покажут боль)

Pinch-zoom закрыт в **SL-3** ✅.
